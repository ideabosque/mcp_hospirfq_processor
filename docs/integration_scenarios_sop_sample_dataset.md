## 2. Purpose and Scope

This SOP defines the ordered live integration scenarios used to validate `mcp_hospirfq_processor` against the local `silvaengine_gateway` route for `rfq_engine`, using **real B2B airline seat data** derived from the Excel file `2026大陸機位B2B動態.xlsx`. All item names, airline names, city names, routes, fares, and cancellation policies are in **Chinese (Traditional)** — reflecting the actual B2B dataset for China-Taiwan airline seat distribution.

Data is sourced from two directories:

- **Product data** (items, provider items, batches, price tiers, bundles, bundle components, cancellation policies, catalog refs, discount prompts) comes from `sample_dataset/` — generated from the real Excel file.
- **Non-product operational data** (segments, segment contacts, seeded requests, seeded quotes, seeded quote items, FX rates) comes from the parent `prepare_test_data/` directory — generated by the existing Faker-based prepare scripts.

The product dataset covers 33 flight routes (items), 42 provider items (airline offerings), 103 provider item batches (specific flight dates with seat counts), 126 item price tiers (adult / child / infant), 12 travel itinerary bundles, 44 bundle components, 1 cancellation policy, and 84 discount prompts (3 global + 6 segment + 33 item + 42 provider-item). The non-product dataset covers 3 segments with 15 contacts, 5 seeded requests, 9 seeded quotes, 5 seeded quote items, and 16 FX rates. All product prices are in **TWD**; non-product operational data uses **USD** as base currency.

- **In scope:** MCP HospiRFQ processor facade, GraphQL client path through the gateway, catalog-first item discovery from the `FLIGHTS` namespace, item/provider lookup, RFQ request lifecycle, quote lifecycle with per-passenger (`per_pax_type`) quote item pricing, pricing (price tiers + discount prompts), installments, files, segments, availability holds, bundles (中國大陸旅遊行程), cancellation policies (經濟艙 團體旅遊退票政策), and catalog inquiry.
- **Out of scope:** production testing, destructive cleanup of generated live test records, cloud provisioning, third-party production side effects, UI testing, load testing, MCP HTTP transport testing (covered by `integration_scenarios_sop_http.md`), direct database connection from the test runner (reconciliation is API-level only), data migration between backends, backend tuning/HA configuration.
- **System(s) under test:** `mcp_hospirfq_processor`, local `silvaengine_gateway`, `rfq_engine`, and catalog/KGE-backed inquiry path reachable through `rfq_engine`.

## 2.1 Controlling End-to-End Testing Procedure

The following procedure is authoritative for this sample-dataset SOP:

1. Confirm the gateway is running and reachable before any test execution (environment validation gate).
2. When the backend is PostgreSQL, confirm the schema is at the latest Alembic revision (`alembic -c migration/alembic.ini upgrade head` applied) before any tool execution (migration readiness gate — dependency state `initialized`).
3. Confirm the prepared **non-product** data fixtures (`segments_and_contacts.json`, `requests.json`, `quotes.json`, `quote_items.json`, `fx_rates.json`) from `prepare_test_data/` have been loaded via their respective prepare scripts.
4. Confirm the prepared **product** data fixtures (`flight_products.json`, `flight_catalog_refs.json`, `discount_prompts.json`) from `sample_dataset/` have been loaded via the sample-dataset prepare scripts.
5. Execute the end-to-end live integration testing with the script `mcp_hospirfq_processor/tests/run_integration.py`, which drives the processor tools in-process through the gateway GraphQL route.
6. Use variables from `mcp_hospirfq_processor/tests/.env` to target the local integration instance. Do not hard-code credentials, endpoint IDs, partition IDs, gateway URLs, or `DATABASE_URL` values in generated reports.
7. Read and use the prepared data in `sample_dataset/` (product) and `prepare_test_data/` (non-product) as the reference dataset and dependency source for function inputs, expected relationships, and scenario ordering.
8. Before any item/request/quote workflow is executed, perform catalog search first (`inquire_catalog` with namespace `FLIGHTS`) and use the catalog result to identify the primary item and provider item for the remainder of the test run.
9. Reconcile the catalog result to prepared data in `sample_dataset/flight_catalog_refs.json`, so the selected catalog node maps back to known `itemUuid`, `providerItemUuid`, airline, route, and bundle reference data.
10. Build the function dependency map before execution, then derive the test sequence priority from that dependency map rather than file-discovery order.
11. Perform end-to-end live integration testing in dependency order.
12. Address any implementation, runner, data-contract, repository, or scenario-ordering issues found during live execution.
13. Retest affected scenarios, then rerun the full dependency-ordered suite until all required calls pass with zero unexpected error responses.
14. Export the final per-function arguments and outputs into the project `docs/` directory as a dated results file.

## 6. Execution Order

```text
catalog_discovery -> items -> requests -> quotes -> pricing -> installments -> files -> segments -> availability -> bundles -> cancellation -> catalog
```

**Reason for deviation from default:** this project is an RFQ processor facade; catalog discovery must identify the primary item before item/provider validation and request creation, request/provider assignment must precede quote creation, quote pricing must precede installment validation, and bundle/policy/catalog readbacks are reference checks. The sequence does not change with backend selection — it only changes where data is persisted.

**Sequence construction rule:** the execution order must be rebuilt or revalidated before each certification run from the actual function dependencies in the runner and codebase. Static file order is not sufficient.

## 8. Failure and Resilience Scenarios

| Scenario | Injected fault | Expected behavior |
|---|---|---|
| missing gateway | stop gateway before live execution | authentication or health validation fails clearly; no tests execute |
| invalid credentials | bad auth env values | token request fails and no tests execute |
| PostgreSQL unreachable (when `DB_BACKEND=postgresql`) | unset `DATABASE_URL` or stop the local Postgres instance | gateway GraphQL calls return a clear persistence error; runner reports blocked; no partial certification |
| schema not at head (when applicable) | skip `alembic upgrade head` | migration readiness gate fails; testing blocked at dependency state `initialized` |
| stale seeded IDs | prepared sample dataset IDs missing from the active backend | affected scenario fails with GraphQL lookup error |
| missing `pax_breakdown` | omit `pax_breakdown` (adult/child/infant) for flight quote item | GraphQL validation rejects `per_pax_type` pricing |
| premature hold expiry | expire a fresh 15-minute hold | expected no-op envelope is recorded |
| batch capacity exhausted | select a batch with `availabilityQty=0` (e.g. `3U3982-20260628-成都`) | availability check returns zero; hold acquire fails with clear capacity error |
